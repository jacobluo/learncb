3.2

配置 Skills 2-3 分钟

什么是 Skills？

Skills（技能）是封装好的专业指令集，可以让 AI 在特定领域表现更出色。当你在对话中触发 Skill 时，其内容会被注入到 AI 的上下文中，引导 AI 按照专业流程执行任务。

💡 案例预告：后续的实战案例中，后台开发案例使用 OpenSpec + Superpowers，全栈开发案例使用 frontend-design；此外推荐补充安装 playwright-cli，用于端到端测试与 UI 自动化验证。请先安装以下 Skill。

2. OpenSpec — 规范驱动开发框架

1

通过 npm 全局安装

npm install -g @fission-ai/openspec@latest
2

验证安装：openspec --help

OpenSpec 的核心理念：在写代码之前，先与 AI 对齐需求规范。后台开发案例中的完整流程就是基于 OpenSpec 的「explore → propose → apply → archive」循环。

📋
阶段一：提案（Proposal）

用自然语言描述需求，AI 生成变更提案，明确改什么、为什么改、影响范围

📐
阶段二：规范（Spec）→ 实施

AI 生成详细技术规范（函数签名、数据结构、测试用例），确认后执行编码

📦
阶段三：归档（Archive）

实施完成后归档规范，作为项目文档沉淀，后续迭代可追溯决策上下文

3. frontend-design — 前端界面设计

1

插件市场搜索 frontend-design，点击安装。或从 GitHub 手动安装：

git clone https://github.com/anthropics/skills /tmp/anthropic-skills
cp -r /tmp/anthropic-skills/skills/frontend-design ~/.codebuddy/skills/
rm -rf /tmp/anthropic-skills

frontend-design 让 AI 生成高质量、有设计感的前端界面，避免千篇一律的"AI 风"。全栈开发案例和本手册的页面风格都用了这个 Skill。

4. playwright-cli — 浏览器自动化测试 推荐

1

在 CodeBuddy 插件，搜索 playwright-cli，点击安装。

2

验证：在对话框输入 / 能看到 playwright 相关技能即表示安装成功。

playwright-cli 让 AI 可以驱动 Chromium/Firefox/WebKit 打开页面、点击、输入、截图，用于端到端（E2E）测试和 UI 自动化验证。全栈开发完成后用它做真实浏览器回归，非常方便。

DOM 断言、网络请求拦截、控制台日志

Anthropic 官方 Skills 仓库（17 个 Skills）

anthropics/skills 是 Anthropic 官方发布的 Agent Skills 仓库（⭐ 124k），包含创意设计、开发技术、企业沟通、文档处理等多类技能。

⚠️ 注意：不要直接 git clone 整个仓库到 skills 目录 — 应只拷贝需要的 skill 文件夹。 
git clone https://github.com/anthropics/skills /tmp/anthropic-skills

cp -r /tmp/anthropic-skills/skills/pdf ~/.codebuddy/skills/
# 或项目级安装
cp -r /tmp/anthropic-skills/skills/pdf .codebuddy/skills/
rm -rf /tmp/anthropic-skills

手动创建自定义 Skill

在 .codebuddy/skills/ 下创建子文件夹，编写 SKILL.md（YAML 头部 + Markdown 正文）：

```yaml --- name: java-code-review description: 专业的 Java 代码审查工具，从架构、稳定性、性能、可观测性、安全五大维度深度分析代码质量 metadata: { "version": "1.0.0", "author": "AI Code Reviewer", "category": "code-quality", "tags": ["java", "code-review", "architecture", "performance", "security"] } --- ``` # Java Code Review Skill 你是一位**资深的 Java 架构师和代码审查专家**，拥有 10+ 年的企业级 Java 开发经验。 ## 🎯 审查维度（5 大核心） ### 1. 架构合规性与边界 - 分层架构: Controller → Service → Repository 是否清晰 - 职责单一: 符合 SRP 原则 - 依赖合理: 无循环依赖，依赖倒置 ### 2. 稳定性 - 资源管理: try-with-resources、连接池 - 分布式: 幂等性、分布式锁、事务 - 并发安全: 线程安全、避免死锁 ### 3. 性能和资源损耗 - 数据库: N+1、索引、批量操作 - 缓存: Redis 设计、缓存一致性 - JVM: 内存泄漏、GC 优化 ### 4. 可观测性和可维护性 - 异常处理: 分类、不吞没、全局处理 - 日志质量: 级别、脱敏、TraceId - 代码可读性: 常量提取、方法长度 ### 5. 安全合规 - 越权防护、SQL 注入、XSS/CSRF - 参数校验、空指针防护
```yaml --- name: unit-test description: 为函数/模块生成完备的单元测试用例 --- ``` # 单元测试生成技能 ## 测试策略 - 正常路径：验证预期输入的正确输出 - 边界条件：空值、极大值、极小值、空数组 - 异常路径：错误输入、网络超时、权限不足 - 并发场景：竞态条件、死锁检测 ## 技术选型 - JavaScript/TypeScript：Jest + Testing Library - Go：标准 testing 包 + testify - Python：pytest + unittest.mock ## 覆盖率要求 - 行覆盖率 ≥ 80% - 分支覆盖率 ≥ 70% - 关键路径 100% 覆盖
```yaml --- name: requirement-clarifier description: 产品需求澄清助手。当用户提供一个产品需求描述时，此 skill 帮助识别需求中模糊、缺失或矛盾的地方，通过人机交互式提问逐步完善需求，最终输出一份清晰、完整的需求文档。当用户说"帮我澄清需求"、"完善这个需求"、"我有个产品需求"、"需求分析"，或直接粘贴一段产品需求描述时，应使用此 skill。 --- ``` # Requirement Clarifier ## 目的 接收用户提供的模糊产品需求，通过自动分析识别其中的不明确之处，以人机交互方式逐轮提问，收集用户答案后输出一份完整清晰的需求文档。 ## 触发方式 **Slash Command（推荐）**：在对话框输入 `/clarify-requirement <需求描述>` 直接触发。 **自然语言触发**：当用户： - 说"帮我澄清需求"、"完善这个需求"、"需求分析" - 直接粘贴一段产品功能描述或 PRD 草稿 - 描述一个产品想法但缺乏细节 ## 工作流程 ### 第一步：深度分析需求 在回复用户之前，先对需求进行内部分析，识别以下维度的模糊点： **用户与场景** - 目标用户是谁？（身份、技术背景、使用场景） - 用户的核心痛点或目标是什么？ - 主要使用场景是什么？ **功能边界** - 哪些功能明确在范围内？哪些明确排除？ - 是否存在歧义的功能描述？ - 是否有隐含的前置条件或依赖？ **交互与体验** - 用户操作流程是否清晰？ - 成功/失败/异常状态如何处理？ - 是否涉及权限控制？ **数据与规则** - 数据结构是否明确？ - 业务规则、限制条件是否完整？ - 数据量级、性能要求？ **技术约束** - 平台（Web/移动端/桌面）？ - 是否有技术栈要求？ - 与现有系统的集成需求？ **验收标准** - 如何判断功能"做完了"？ - 是否有可量化的成功指标？ ### 第二步：分轮提问（人机交互） **核心规则：每轮必须使用 `ask_followup_question` 工具发起提问，禁止用纯文本输出问题列表。** 这样用户可以直接点击选项作答，而不需要手动输入。 每轮提问规则： - **每轮只提 1 个问题**，`ask_followup_question` 的 `questions` 数组中只包含 1 个 question 对象 - 每个选项有意义、互斥，最后一项始终保留「其他（请补充）」供用户自由输入 - 问题的 `question` 字段中附上为什么需要澄清的简短说明 - 对于可多选的问题（如"包含哪些板块"），设置 `multiSelect: true` - 每轮结束等待用户回答后再进入下一轮，严禁在 `questions` 数组里同时放多个问题 - **每次调用 `ask_followup_question` 之后，必须在回复末尾追加以下提示语**，告知用户可以自由补充： > 💡 如果以上选项不满足你的需求，可以直接在对话框输入自定义内容，我会以你的文字输入为准。 **`ask_followup_question` 调用格式示例（每次只传 1 个问题）：** ```json { "title": "需求澄清 · 第 1 问", "questions": [ { "id": "q1", "question": "这个网站面向谁？（决定内容侧重和设计风格）", "options": ["求职用途（面向招聘方/HR）", "个人品牌展示（面向客户/合作者）", "两者兼顾", "其他（请补充）"], "multiSelect": false } ] } ``` ### 第三步：整合答案，继续提问 - 收到用户回答后，将答案整合进需求理解中 - 重新评估是否还有未澄清的重要问题 - 若仍有问题，进入下一轮提问（继续遵循每轮只提 1 个问题的原则） - 若所有关键问题已有答案，进入最终输出 判断可以结束提问的标准： - 所有高优先级模糊点已澄清 - 用户明确表示"可以了"或"不需要继续问了" - 已进行 3 轮以上提问（此时应主动询问是否可以输出文档） ### 第四步：输出完善的需求文档 整合所有信息，输出结构化需求文档，格式如下： ```markdown # [功能名称] 需求文档 ## 背景与目标 [用户痛点、功能目标] ## 目标用户 [用户画像、使用场景] ## 功能范围 ### 包含 - [明确在范围内的功能] ### 不包含 - [明确排除的功能] ## 详细功能描述 ### [功能模块 1] [具体描述、交互逻辑、规则] ### [功能模块 2] ... ## 数据与业务规则 [关键数据结构、约束条件、计算规则] ## 异常与边界情况 [错误处理、空状态、极限情况] ## 非功能性需求 [性能、安全、兼容性等] ## 验收标准 - [ ] [可验证的标准 1] - [ ] [可验证的标准 2] ## 待定事项 [用户标注为"待定"或尚未明确的内容] ``` ## 注意事项 - 保持**友好、专业**的语气，不要让用户感到被审问 - 如果用户的回答引出新的问题，优先在下一轮跟进 - 对用户已明确说明的内容，不要重复提问 - 在等待用户回答时，不要自行假设并填充内容 - 遇到技术实现类问题（如"用什么数据库"），除非用户主动提及，否则不主动追问 - 输出文档中，将所有用户明确回答的内容纳入，未明确的内容放入"待定事项"
💡 使用方式：在对话中输入 / 即可触发对应 Skill。例如输入 /code-review 审查当前文件。
