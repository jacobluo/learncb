5.1

实战：用 OpenSpec + Superpowers 开发 Java 秒杀 API

项目背景

本案例完整记录了使用 CodeBuddy 从零开发一个秒杀商品 API 后端服务的全过程。项目需要处理高并发库存扣减、订单创建、防超卖等核心问题。开发过程中综合运用了前面章节介绍的 Skills、Rules 等能力，展示 AI 编程的工程化实践。

代码仓库：https://cnb.cool/relaxorg/java-oneshot-svr

Spring Boot 3.x MyBatis-Plus Redis Lua MySQL Docker Compose TDD

前置准备：安装 Skills 和配置 Rules

本案例使用了两个 Skill 插件和两条 Rule，需要提前配置好（参考 三-2 配置 Skills 和 三-5 配置 Rules）：

类型名称说明
SkillOpenSpec规范驱动开发框架，提供 explore → propose → apply → archive 全流程
SkillSuperpowers工程方法论技能包，提供 brainstorming、writing-plans、test-driven-dev、code-review 等
Rulejava-coding-styleJava 编码规范（命名、分层、异常处理等），设为 Always 生效
Rulekarpathy-guidelinesKarpathy 风格编程准则（简洁、可读、避免过度抽象），设为 Always 生效

OpenSpec 安装后提供以下斜杠命令：

命令阶段说明
/opsx:explore探索AI 主导需求探索，逐项提问确认技术决策
/opsx:propose提案生成变更提案（proposal + design + specs + tasks）
/opsx:apply实施按 tasks 逐步生成代码，可配合 Superpowers 的 TDD 模式
/opsx:archive归档归档变更记录 + 同步 specs

AGENTS.md — 项目规范

自定义 AGENTS.md 规范开发流程

在项目根目录创建 AGENTS.md，定义项目的技术栈、架构约束和开发工作流。CodeBuddy 在每次对话中会自动读取该文件，确保 AI 遵循项目约定。配合 java-coding-style 和 karpathy-guidelines 两条 Rule，形成完整的规范体系。

本案例使用的 AGENTS.md 完整内容：

```markdown # java-oneshot-svr 使用 Spring Boot 开发一个秒杀商品的 API 后端服务，需要处理高并发库存扣减、订单创建、防超卖等核心问题。采用 OpenSpec（规范驱动开发）+ Superpowers（工程方法论）组合 **产品与技术细节** → `openspec/AGENTS.md` **活跃提案与规格** → `openspec list` / `openspec/changes/` / `openspec/specs/` --- ## 开发工作流 **OpenSpec 三段 + Superpowers 纪律**：OpenSpec 管规划，Superpowers 管编码纪律。 1. 新功能一律走 `/opsx:propose`，**必须**进行propose前调用 skill `brainstorming` 进行脑暴 2. 使用 `/opsx:apply` 实施任务时，必须启用 `test-driven-development`：先写失败测试，再写实现 3. `tasks.md` 中存在 2+ 个独立、无依赖的任务时，必须启用 `dispatching-parallel-agents` 并行派发 4. 完成主要步骤后必须请求 `code-reviewer` 评审，声明「完成」前必须执行 `verification-before-completion` 5. 遇到测试失败、异常行为时必须启用 `systematic-debugging`，禁止直接猜测原因 6. 每个新功能必须在独立 Git Worktree 中开发，禁止直接在 main 分支修改代码 ### Superpowers 技能触发规则 | 触发场景 | 必须调用的技能 | |---------|--------------| | 启动新功能开发 | `using-git-worktrees` | | 实现新功能 / 修 bug | `test-driven-development` | | 遇到测试失败、异常行为 | `systematic-debugging` | | 声称"完成"之前 | `verification-before-completion` | | 2+ 个独立可并行的任务 | `dispatching-parallel-agents` | | 完成主要步骤或重大改动 | 请求 `code-reviewer` 评审 | ### 规范约束 - `proposal.md` 生成后必须人工检查 Out of Scope - `tasks.md` 生成后必须人工检查任务顺序与遗漏 - Spec 描述行为（GIVEN/WHEN/THEN），不描述实现 - 功能完成后必须执行 `/opsx:archive` ```
阶段一

需求探索（opsx:explore）

@command://opsx:explore 使用 Spring Boot 开发一个秒杀商品的 API 后端服务，需要处理高并发库存扣减、订单创建、防超卖等核心问题
AI 分析系统架构

CodeBuddy 自动分析秒杀系统的四层架构（流量管控 → 库存预判 → 订单创建 → DB 落库）和 5 个关键决策点（库存策略、技术栈、防超卖、API 设计、数据模型）。

逐项提问确认技术选型

AI 针对每个技术选型提供 A/B 选项，用户快速选择确认。共 11 轮 Q&A，覆盖项目定位、ORM、MQ、认证、限流、数据库等全部决策。

数据库设计深入探讨

逐表逐字段确认：user 表极简、seckill_goods 靠时间判断状态不存 status、订单冗余商品快照、UNIQUE 防重复下单。

Redis 和 Lua 脚本设计

确认 Redis Key 命名、Lua 脚本职责（库存扣减 + 防重复一体化）、DB 事务失败后 Redis 立即回补策略。

探索完成 — 输出架构全景

技术决策汇总 + 架构全景图 + 防超卖三道防线（Redis Lua → DB 乐观锁 → UNIQUE 索引）。所有决策自动沉淀到 AGENTS.md。

技术决策汇总

决策项选择结论
项目定位A教学/演示级
ORMAMyBatis-Plus（精确控制 SQL）
MQA不引入，同步下单
认证B简单 JWT（带角色）
限流AGuava RateLimiter + 自定义注解
数据库AMySQL (Docker)
RedisADocker 真实 Redis
重复购买A不允许（一人一单）
Spring BootA3.x + Java 17
构建工具AMaven
Docker ComposeA提供一键启动
阶段二

Spec 1：项目骨架搭建

propose — 创建变更提案

输入 /opsx:propose 项目骨架搭建，自动生成 proposal + design + 6 个 spec + tasks（8 组 22 个任务）。涵盖统一响应、JWT 认证、限流、数据层、Redis、Docker 基础设施。

提案生成完成

列出所有 Artifact：proposal.md（Why/What）、design.md（7 个技术决策）、6 个 spec 目录、tasks.md（22 个任务）。请人工检查后执行。

apply — 执行实施

输入 /opsx:apply，CodeBuddy 按 tasks.md 顺序逐步创建代码：项目结构 → 统一响应 → JWT → 限流 → 数据层 → Redis → 拦截器 → 测试。

实施进行中

自动创建实体类、Mapper、配置文件、种子数据（BCrypt 真实密码 hash）、Maven Wrapper 等。遇到问题自动修复。

实施完成 — 26/26 tasks ✓

全部任务完成，4 个集成测试通过（健康检查、登录、无 Token 拒绝、角色鉴权），编译 0 错误 0 lint。

archive — 归档

输入 /opsx:archive，归档到 archive/ 目录，6 个 capability spec 同步到 openspec/specs/。完整决策链路可追溯。

阶段三

Spec 2：秒杀核心功能（TDD 模式）

propose — 秒杀核心提案

输入 /opsx:propose 秒杀核心功能，scope 选择 B（秒杀下单 + 商品查询），生成 5 个 spec + 18 个任务，含 3 个并发测试场景。

提案完成 — 5 个 spec

seckill-execute（Lua 脚本 + DB 事务 + Redis 回补）、stock-init（库存初始化）、goods-query（商品查询）、concurrency-test（3 并发场景）、data-layer（decrStock）。

apply + TDD — git worktree 隔离开发

输入 /opsx:apply 并要求按 Superpowers 的 test-driven-development 模式。使用 git worktree 创建 feature/seckill-core 分支隔离开发。

TDD 循环：RED → GREEN

RED Phase：先写 8 个测试（5 功能 + 3 并发），确认全部失败。GREEN Phase：实现 SeckillService + GoodsService + 3 个 Controller，测试从红变绿。

实施完成 — 12/12 tests ✓

18/18 任务完成。12 个测试全部通过（4 骨架 + 5 功能 + 3 并发：防超卖 ✓、防重复 ✓、时间窗 ✓）。

合并 & 归档

feature/seckill-core 合并到 master（--no-ff），推送到远端，归档 spec + 清理 worktree。Git 历史清晰可追溯。

总结

完整开发流程回顾

阶段命令产出
初始化openspec init生成 .openspec/ 目录 + AGENTS.md
需求探索/opsx:explore11 项技术决策 + 架构全景 + 数据库设计

Spec 2 归档/opsx:archive合并到 master + 推送远端
⚠️ 关键点：propose 后必须人工 Review
/opsx:propose 生成的 proposal.md、design.md、specs/、tasks.md 都是 AI 草案，在执行 /opsx:apply 之前必须由开发者人工审阅。如果发现不符合要求（如 scope 偏差、任务遗漏、技术方案不合理），可以直接修改这些文件后再执行实施。AI 生成的规范是起点，不是终点——人始终掌握最终决策权。
💡 关键收获：OpenSpec 的「explore → propose → apply → archive」循环提供结构化开发流程；Superpowers 的 test-driven-dev 保证代码质量；java-coding-style 和 karpathy-guidelines 两条 Rule 让 AI 生成的代码自动遵循团队规范。三者组合，让 AI 编程从"随便写写"变成可追溯的工程化开发。
